Administración de la BD

Para llevar a cabo la Administración de la BD Oracle del Simulador Táctico Galeón, se suministran diferentes Scripts SQL que facilitan este trabajo. Para la ejecución de éstos scripts la Herramienta de Preparación del Simulador Táctico Galeón incluye una utilidad de para la ejecución de scripts SQL contra el SGBD Oracle (ver Manual de Preparación – Descripción General). Los scripts se instalan en el directorio “C:\Archivos de Programa\Galeon\data.bd\Scripts”.

Las acciones más comunes de Administración de la BD del sistema son las que se describen paso a paso en las siguientes tablas. En la última columna se indican los scripts que deben ejecutarse para realizar los pasos requeridos.

Creación de BD nueva sobre usuario nuevo:

Paso Acción Script SQL
1 Crear nuevo usuario tipo glndbCreateUserType
2 Crear esquema de BD glndbCreateSchemaOra

Creación de BD nueva sobre usuario existente:

Paso Acción Script SQL
1 Borrar esquema de usuario tipo existente glndbDeleteSchemaOra
2 Crear esquema de BD glndbCreateSchemaOra

Borrado de BD:

Paso Acción Script SQL
1 Eliminar el usuario y todos los objetos que posea glndbDropUser

Creación de usuarios de sólo lectura, de preparación de juegos o de acceso total para una BD existente:

Paso Acción Script SQL
1 Conceder al usuario que comparte la BD los permisos necesarios para compartir sus objetos glndbUserForShareGrant
2 Crear sinónimos públicos para los objetos compartidos glndbCreateSynonyms
3 Crear nuevo usuario glndbCreateUserType
4

Asignar al nuevo usuario los permisos necesarios para acceder a los objetos del usuario que los comparte:

sólo lectura

preparación de juegos

acceso total

glndbCreateUserRead

glndbCreateUserPrep

glndbCreateUserWrite
5 Eliminar los permisos al usuario que comparte los objetos (este paso es opcional) glndbUserForShareRevoke

Creación de usuario de login:

Paso Acción Script SQL
1 Crear usuario para login (sólo es necesario hacerlo al instalar por primera vez, para configuración los datos de conexión del sistema) glndbCreateUserLogin

Actualización de esquema de BD:

Paso Acción Script SQL
1 Actualizar el esquema (sólo para instalación de nuevas versiones del sistema) NN-glndbOraUpdate[d1.m1.a1][d2.m2.a2].sql

Réplicas de BD: para replicar BD deben emplearse las utilidades de Importación y Exportación incluidas en la Herramienta de Preparación (ver Manual de Preparación – Descripción General), siguiendo los siguientes pasos.

Paso Acción
1 Exportar la BD del usuario que se desea replicar.
1 Importar con el nuevo usuario la BD exportada previamente.

glndbDropUser

-- ****************************************************************************
--
-- Simulador Táctico Galeón
--
-- Eliminación de un usuario.
--
-- Este script debe ejecutarse desde un usuario con privilegios suficientes
-- para efectuar las operaciones que se indican en él. El usuario puede ser
-- el usuario system de Oracle.
--
-- ****************************************************************************
-- Se deben sustituir las siguientes palabras:
--
-- USERNAME:
-- Nombre de usuario que se quiere eliminar.
--
-- USER_ASSOCIATED_ROLE:
-- Nombre del role asociado al usuario.
--
-- Según la política de usuarios y permisos seguida en Galeón, pueden existir
-- diferentes conjuntos de objetos, normalmente tablas, (que se corresponden
-- con lo que podríamos denominar diferentes bases de datos) cada uno de ellos
-- con un usuario propietario que tiene acceso total a los mismos. Se definen
-- posteriormente dos o más usuarios a los que el propietario de los objetos
-- concede determinados permisos sobre sus objetos (consulta o modificación
-- y creación restringida). Estos permisos pueden ser concedidos directamente
-- a un usuario concreto o asignados a roles y después asignar el role a uno
-- o más usuarios. Usar roles es un procedimiento más general ya que los
-- permisos se asignan al role una vez y luego basta con asignar el role a
-- todos los usuarios a los que se quiera dar los mismos permisos. Normalmente
-- en Galeón sólo va a existir un usuario por role. El concepto de role pierde
-- así toda su utilidad, pero se usa esta forma de asignar permisos por ser
-- un procedimiento más genérico.
--
-- Como los roles se definen a nivel de motor de base de datos, puede haber
-- conflictos al crear roles con el mismo nombre para distintos usuarios que
-- van a acceder a distintos conjuntos de objetos (diferentes bases de datos).
-- Puesto que cada role va a estar asociado sólo a un usuario debido a la
-- forma de proceder en Galeón, al eliminar un usuario es conveniente eliminar
-- también su rol asociado para evitar posibles conflictos de nombres. Por lo
-- tanto es recomendable cuando se crea un role, asignarle un nombre que lo
-- asocie de forma inequívoca con un usuario concreto, y seguir siempre el mismo
-- criterio de asignación de nombres a roles para usuarios. De esta forma se
-- facilita la eliminación del role al eliminar el usuario.
--
-- Tener en cuenta, si el rol está concedido a varios usuarios, que cuando se
-- elimina un role, Oracle lo revoca de todos los usuarios a los que se les
-- había concedido y se elimina el rol a nivel de motor de base de datos.
--
-- ****************************************************************************

glndbUserForShareGrant

-- ****************************************************************************
-- Simulador Táctico Galeón

-- Asignación de permisos de creación de sinónimos públicos, creación de

-- Usuario SYSTEM de Oracle.
--
-- ****************************************************************************

--
-- ****************************************************************************

-- ****************************************************************************

glndbCreateSynonyms

-- ****************************************************************************
-- Simulador Táctico Galeón
--
-- Creación de sinónimos públicos para los objetos de un usuario llamado
-- USER_SHARED, que compartirá sus objetos con otros usuarios. Esto permite
-- el acceso a los objetos por parte de cualquier usuario que tenga permisos
-- sobre ellos calificándolos sólo mediante el nombre del sinónimo.
--
-- ****************************************************************************
-- EJECUTAR DESDE:
--
-- El usuario que va a compartir sus objetos: USER_SHARED
--
-- ****************************************************************************
-- SUSTITUIR:
--
-- USER_SHARED:
-- Nombre del usuario que quiere compartir sus objetos. Este usuario debe
-- tener el mismo esquema que se indica aquí. En caso contrario se deben
-- eliminar o añadir las referencias a los objetos que correspondan. Debe
-- tener concedido el privilegio de creación sinónimos públicos, como indica
-- la sentencia SQL siguiente (este privilegio se lo debe conceder el usuario
-- SYSTEM de Oracle previamente):
--
-- GRANT CREATE PUBLIC SYNONYM TO USER_SHARED;
--
-- Una vez finalizada la ejecución del script, el privilegio de creación
-- de sinónimos públicos puede revocarse mediante la sentencia indicada
-- abajo desde el usuario SUSTEM de Oracle:
--
-- REVOKE CREATE PUBLIC SYNONYM FROM USER_SHARED;
--
-- ****************************************************************************

-- eliminar o añadir las referencias a los objetos que correspondan. Debe

-- GRANT CREATE ROLE TO USER_SHARED;
